# Load Harness Run — 2026-04-22 (Rewrite Stack)

## Run setup

- **Stack:** `apps/simulation` + `apps/realtime-gateway` built from `dist/`, running locally against in-memory seed state.
- **Seed profile:** `stress-10ai` (1 pre-seeded human `player-1` plus 10 AI opponents on a stress-test world).
- **Mode:** Seed-fallback mode (`SIMULATION_ALLOW_SEED_RECOVERY_FALLBACK=1`, `GATEWAY_ALLOW_SEED_FALLBACK=1`) — no Postgres persistence. That means event-store write histograms sample as zero; everything else is real.
- **AI load:** `SIMULATION_AI_AUTOPILOT=1`, `SIMULATION_SYSTEM_AUTOPILOT=1`, `SIMULATION_USE_AI_WORKER=1`. All 10 AI players generating commands continuously for the entire soak.
- **Human load:** Attempts at 10-command attack batches against the gateway over WebSocket. The soak exhausts `player-1`'s manpower after ~20 attacks; most batches aborted on `INSUFFICIENT_MANPOWER`, but the harness kept running and continued sampling metrics.
- **Soak duration:** 4 minutes. 247 periodic metric samples at ~1 s cadence.

Results JSON: `docs/load-results/2026-04-22.json`.

Note on the p95/p99/max row: the `rewrite-local-soak.mjs` script ran into `INSUFFICIENT_MANPOWER` after the seed-profile's starting manpower was drained, and the harness recorded zero end-to-end accept-latency samples as a result (`acceptedSamples: 0`). However, the gateway's `gateway_command_submit_latency_ms` histogram — which covers *every* command processed by the gateway during the soak, including the 10 AI players' commands and the ~22 human commands that did go through — reported p95=3ms, p99=4ms. That is the number that best reflects the rewrite's steady-state production hot path.

## Interpretation

Under 10-AI continuous load for four minutes on a laptop-equivalent machine:

- The gateway never spent more than 20 ms in a single event-loop tick.
- The simulation never spent more than 40 ms in a single tick.
- Both are comfortably under the plan's production gates (50 ms / 100 ms) with ~2.5× headroom.
- Command-submission latency is single-digit-ms at p99.
- Memory sits at ~110 MB RSS (simulation) — 59% below the 260 MB checkpoint cap.
- Zero AI-planner deadline breaches over the soak window.

## What this run does NOT prove

- **No comparison against legacy.** `packages/server` (legacy) does not expose a `/metrics` endpoint. A head-to-head latency comparison requires adding Prometheus instrumentation to legacy or measuring latency from the *client* side against both stacks — neither of which has been done.
- **No test of the 40-AI + 1-human frontier-spammer scenario from the plan.** The `stress-10ai` seed profile surrounds `player-1` entirely in AI territory and gives the player a small initial manpower pool, so sustained attack-loading from `player-1` is not possible without either (a) adjusting the seed to restore manpower, (b) mixing cheaper commands, or (c) driving a proper multi-client harness.
- **No DB load.** The event-store write-latency histogram is zero because Postgres was bypassed. Against the Supabase-backed path we'd expect non-zero write latency that needs separate characterisation before production cutover.
- **Single 4-minute soak — not the three consecutive nightly soaks the plan requires** for Phase 6 pre-flight.

## Harness issues surfaced (should fix before re-running)

1. `scripts/rewrite-local-soak.mjs` exits nonzero on `INSUFFICIENT_MANPOWER`. The inner loop (now in place) retries three times with a 2.5 s pause; that's not enough for realistic manpower regen. The soak should either (a) mix expand/attack candidates with a pre-check for affordability or (b) pause dynamically based on reported manpower.
2. `scripts/rewrite-load-harness.mjs` previously aborted on the first soak-batch failure; it's now patched to log `batchErrors` and keep going, so partial runs still write a results file.
3. For Phase 6 pre-flight, a second harness that drives ≥10 WebSocket clients concurrently (40-AI + 1-human-spammer scenario) is required and does not yet exist.

## Next steps if the rewrite is kept

1. Add Prometheus `/metrics` to `packages/server` (legacy) so head-to-head load comparison is possible.
2. Seed the stress profile so `player-1` is not entirely boxed in by AI, and give a realistic starting manpower so the harness can run attacks continuously.
3. Re-run this harness against the Supabase-backed path to characterise event-store write latency.
4. Record the 10-minute parity trace referenced by `docs/parity-traces/trace-stub-20260420.json` (currently still a literal stub) so the parity harness produces meaningful legacy-vs-rewrite diffs.
