### Fix #6 — Vision expansion cache (`classifyVisibilityForPlayer`)

- **Where:** `apps/simulation/src/runtime.ts:2657-2773`, inner stamp loop at 2680-2698.
- **Cost evidence:** for each `exportVisibleStateForPlayer` call, for the primary player + each ally, iterates `summaryForPlayer(id).territoryTileKeys` and stamps `(2r+1)²` neighbors into a Set. Typical `r = 4-6` ⇒ 81-169 stamps × O(thousands of tiles). No cache. Sibling `exportBarbActivationVisibleUnion` (2798-2828) already proves the cache shape works.
- **Approach:** new `apps/simulation/src/vision-expansion-cache.ts` exposing `getOrComputeExpansion(playerId, territoryTileKeys, vision, visionRadiusBonus, version) → Set<string>`. Signature = `${tileCollectionVersion}:${vision}:${visionRadiusBonus}`. Invalidate via the existing `plannerPlayerTileCollectionVersionByPlayer.set` call sites; also drop entry on player elimination. `classifyVisibilityForPlayer` becomes a composer that pulls primary + ally sets from cache.
- **Files:** new `vision-expansion-cache.ts` (~120 lines). Edit `runtime.ts` only at the entry points (replace inline `addVision` body with cache lookup; wire `clear(playerId)` into elimination path).
- **Determinism:** Set iteration order is insertion order in V8; only used through `.has()` / union ⇒ no order leak. Verify by snapshot test.
- **Risk:** stale cache on a tile-collection update that didn't bump version. Mitigation: extend existing unit tests in `runtime.test.ts` to cover ownership change → re-export → expect new key set; new test covering vision-mod change.
- **Acceptance:**
  - Unit: `vision-expansion-cache.test.ts` — hit/miss/invalidate semantics + ally composition.
  - Bench: new `vision-expansion-cache.bench.ts` with 5k-tile player; warm ≥ 5× faster than cold; cold within 2× of inline.
  - Integration: prod-shape-load p99 < 35 ms (was 43.7 ms in PR #397); max-block < 180 ms (was 220 ms) — `runtime.prod-shape-load.test.ts` thresholds.

### Fix #7 — `incomePerMinuteForPlayer` fast-path

- **Where:** `runtime.ts:3164-3170` calls `buildPlayerUpdateEconomySnapshot` (`player-update-economy.ts:367-519`). Reads only `.incomePerMinute`. Snapshot allocates 12 Maps + does 6 sorts.
- **Call sites (verified):** `runtime.ts:2355, 2626, 2920, 3034, 3192, 5347, 5404`. In a 5-player game broadcasting per-player visible state every tick, this is 25+ calls per tick.
- **Approach:** new `apps/simulation/src/player-economy-fast-income.ts` exposing `incomePerMinuteOnlyForPlayer(player, summary, tiles, dockContext) → number`. Walks settled-towns (using `firstThreeTownKeys` + `fedTownKeys` already computed by helpers) and docks; sums gold sources only. Reuse `townGoldPerMinuteForPlayer` so the formula stays in one place. `buildPlayerUpdateEconomySnapshot` calls the same primitives so they stay byte-equal.
- **Files:** new `player-economy-fast-income.ts` (~120 lines). `runtime.ts` swap the 7 call sites. Keep `buildPlayerUpdateEconomySnapshot` untouched.
- **Determinism:** must call `applyManpowerRegen` / not call it identically to current path. Audit each of the 7 sites; the snapshot path doesn't apply manpower regen, neither should the fast path.
- **Risk:** drift between fast path and snapshot. Mitigation: equality test that runs both on 50 fixture players and asserts `fast === snapshot.incomePerMinute` to 4 decimal places (the rounding tolerance the snapshot uses).
- **Acceptance:**
  - Unit: equality test ≥ 50 randomized players from a prod-shaped fixture.
  - Bench: fast path ≥ 20× faster than snapshot path on a 200-tile player; ≥ 10× on a 5000-tile player.

### Fix #8 — Memoize `settledTiles` / `orderedSettledTownTiles` per player

- **Where:** `player-update-economy.ts:377-382` (`settledTiles`, `orderedTownTiles`), :283-286 (`ownedSettledTowns` inside `buildFedTownKeys`).
- **Cost evidence:** every snapshot call does `[...summary.territoryTileKeys].map(tiles.get).filter(...)`. Called per player per exportState plus inside `emitPlayerStateUpdate` (5347, 5404 already counted in fix #7). With fix #7, this is reduced for the income-only path — but full snapshot in `emitPlayerStateUpdate` still pays the cost on every tech/domain/player update.
- **Approach:** lift into the existing per-player summary cache (find via `summaryForPlayer` definition). Add `settledTileList: readonly DomainTileState[]` and `orderedSettledTownTileList: readonly DomainTileState[]`. Bumped by the same dirty-pipeline that bumps `tileCollectionVersion`. Update `buildPlayerUpdateEconomySnapshot` and `buildFedTownKeys` to accept pre-filtered lists (preserve current signature with optional argument for backward compat in tests).
- **Files:** edit `player-runtime-summary.ts` (or wherever the summary cache lives — locate via `grep -n "PlayerRuntimeSummary" apps/simulation/src`); edit `player-update-economy.ts`. Both should stay under 500 lines.
- **Determinism:** pre-filtered lists must keep stable ordering. Use `summary.territoryTileKeys` iteration order, which is already deterministic (Set insertion). Add a golden test on a fixture.
- **Risk:** "is settled" predicate has subtleties (decay, fort-takeover). Mitigation: golden test of `settledTileList` after each tile-state mutation primitive — replaceTileState, decay tick, fort flip.
- **Acceptance:**
  - Unit: invalidation/freshness tests for each tile-state mutation primitive.
  - Bench: extend `player-economy-fast-income.bench.ts` with full-snapshot path; expect ≥ 1.5× speedup on the 5000-tile player.

### Fix #9 — Pool bucket Maps in `buildPlayerUpdateEconomySnapshot`

- **Where:** `player-update-economy.ts:386-398` (12 `new Map()`); `:475-483` (`[...sink.values()].reduce`); `:502-507, 510-516` (`sortedBuckets` allocates `[...map.values()].sort(...)`).
- **Cost evidence:** every snapshot call. Sim is single-threaded ⇒ pooling is safe.
- **Approach:** module-level pool: `const bucketPool: Array<Map<string, EconomyBucket>>` with `acquire()` / `releaseAll()`. Snapshot acquires 12 at the top, releases all at the bottom. `sortedBuckets` rewrites to write into a pre-allocated array and sort in place; the returned array is then handed off to the snapshot (and the pool acquires fresh arrays next call). Add explicit comment that the function is non-reentrant.
- **Files:** edit `player-update-economy.ts` only. If it grows past 500 lines, split bucket-pool helpers into `player-economy-bucket-pool.ts`.
- **Determinism:** Map insertion order is preserved across `clear()` — verify the sort comparator (line 105) is stable enough that pooling doesn't change tie-breaking. Comparator is `(amt desc, label asc)` ⇒ already total ⇒ no order leak.
- **Risk:** reentrancy if the snapshot is ever called recursively (e.g. from a callback). Today it isn't — guard with a `inUse` flag that throws in dev to catch future misuse.
- **Acceptance:**
  - Unit: golden test comparing pooled vs non-pooled output across 50 fixture players; must be byte-equal via `JSON.stringify`.
  - Bench: ≥ 15% wall-clock reduction on `buildPlayerUpdateEconomySnapshot` for a 5000-tile player.

### Fix #10 — `exportState` tile projection — single-pass, drop spread

- **Where:** `runtime.ts:2575-2601`.
- **Cost evidence:** `[...this.tiles.values()].map(...).sort(...)` allocates the values array + the map output array; the per-tile spread `...(cond ? {x} : {})` runs ~14 conditionals creating tiny objects. With ~600k tiles, that's millions of micro-allocs per export.
- **Approach:** extract to `apps/simulation/src/export-state-tile-projection.ts`: `projectTilesForExport(tiles: ReadonlyMap, cache: TileDeltaStringifyCache) → readonly TileExportRow[]`. Pre-size `new Array(tiles.size)`. Single `for…of`. Plain `if (cond) entry.field = …`. Sort the populated array in place by `(x asc, y asc)`.
- **Files:** new `export-state-tile-projection.ts` (~80 lines). Edit `runtime.ts` to import + call.
- **Determinism:** must produce byte-equal output to current implementation. Golden test (see acceptance).
- **Risk:** subtle behavior of conditional spreads vs `if` assignment — e.g. `...(tile.resource ? { resource: tile.resource } : {})` only adds the key when truthy; `if (tile.resource) entry.resource = tile.resource` does the same. Confirm none of the source fields are `0`/`""` cases where falsy ≠ empty. Verify by golden test.
- **Acceptance:**
  - Unit: golden test on a 500-tile fixture asserting `JSON.stringify(new) === JSON.stringify(old)`.
  - Bench: ≥ 30% faster on a 600k-tile world (synthetic fixture; load via `__bench__/load-prod-snapshot.ts`).
